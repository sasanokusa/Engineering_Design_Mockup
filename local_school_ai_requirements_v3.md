# 校内機密データ向けローカルAI支援基盤 要件定義 v3

## 1. 概要
本システムは、校内で扱う機密性の高い情報を外部クラウドへ送信せず、校内または研究室内のローカル計算資源上で安全に処理するためのローカルAI支援基盤である。

対象とする情報には、保護者面談記録、各種相談記録、学内限定資料、学生の論文・レポート・提出物などが含まれる。

本システムは単一の用途に閉じたツールではなく、以下を統合的に扱える基盤を目指す。

- 学内資料を参照した質疑応答
- 論文・レポート・提出物の読解支援
- 複数文書の比較支援
- 音声からの議事録作成支援

ただし、全機能を一つの同一UIフローに無理に統合するのではなく、**共通チャット基盤**と**専用ツール群**を組み合わせる構成を採用する。

---

## 2. 背景
教員へのヒアリングにより、以下のような需要が確認された。

1. 保護者面談や各種相談において、録音や記録を安全に処理し、議事録案を作成したい。
2. 先進的・未公開の内容を含む学生の論文や研究レポートを既製のクラウドAIへ送ることを避けたい。
3. 学内から外部に出してはならない資料を参照しながら回答を作成したい。
4. 個人情報を含むレポートや提出物であっても、外部流出の懸念なくAI支援を受けたい。
5. レポートの流用・写しの疑いを確認するため、複数ファイルの差分や類似箇所を比較したい。

従来のクラウドAIサービスは利便性が高い一方、入力データの取り扱いに対する不安が残る。そのため、**ローカル環境で完結するAI基盤**を整備する意義が大きい。

---

## 3. 目的
本システムの目的は以下の通りである。

- 校内機密データを外部クラウドへ送信せずにAI支援を実現すること
- 教員が日常業務の中で使いやすい形でAI機能を提供すること
- 単発のPoCではなく、将来的に拡張可能な基盤として設計すること
- 文書参照、文書比較、議事録生成など異なる用途を、共通基盤の上で整合的に運用できるようにすること

---

## 4. スコープ

### 4.1 本システムで扱う主要機能
本システムは以下の4系統を扱う。

1. **資料参照チャット**
   - 学内資料や登録済み文書群を参照しながら回答する
2. **論文・レポート読解支援**
   - 単一または複数の提出物をその場で読み込ませ、要約・添削・確認を行う
3. **文書比較支援**
   - 複数ファイル間の差分、類似箇所、流用疑い箇所を確認する
4. **議事録作成支援**
   - 音声から文字起こしを行い、整形後に議事録案を作成する

### 4.2 本システムで重視する点
- 機密保持
- ローカル実行
- モデル差し替えの容易さ
- UIの使いやすさ
- 将来拡張時の整合性

---

## 5. 基本方針

### 5.1 中心UIはチャット型とする
資料参照や文書読解、比較支援など、今後の用途拡張を考えると、中心UIは一般的なチャットエージェント型が最も扱いやすい。

ただし、チャット本体は以下の責務に限定する。

- 会話履歴の管理
- ツール呼び出し
- 結果の表示と整理

重い文書処理や音声処理は、チャット本体ではなく専用ツールまたはバックエンドサービスが担当する。

### 5.2 議事録は専用ツールとして扱う
議事録作成は、通常の文書参照チャットとは性質が異なる。

- 音声入力を起点とする
- ASR → 整形 → 議事録生成の多段処理が必要
- 長時間ジョブになりやすい
- raw transcript と整形済みテキストと議事録案を見比べたい

このため、議事録機能は**独立した専用ツール**として実装する。
ただし、チャットUIのツール起動メニューや「+」ボタン相当から呼び出せる設計とする。

## 6. システム構成
本システムは以下の論理コンポーネントで構成する。

### 6.1 共通チャット基盤
役割:
- 会話UI

- セッション管理
- ツール呼び出し
- 応答整形

対象例:
- 学生生活ハンドブック
- 校則
- 学内限定資料

- FAQ元資料

### 6.3 一時文書基盤
役割:

対象例:
- 論文原稿
- レポート

### 6.4 文書比較ツール
役割:
- 複数文書の差分表示

## 7. モデル・推論方針

### 7.1 ベースLLM
ベースLLMは **Gemma 4 31B** を基本候補とする。
理由は以下の通り。

- ローカル実行可能な高性能モデルであること
- 長文文書処理や文書参照に十分な性能が期待できること
- 校内機密データ用途において、信頼性と説明性のあるベースモデルを採用しやすいこと

### 7.2 モデル差し替え方針
アプリケーション本体はモデルに直接依存せず、OpenAI互換APIや抽象アダプタ経由で呼び出す。
これにより、以下の差し替えを容易にする。

- Gemma 4 E4B → Gemma 4 31B
- Gemma 系 → 他の信頼できるローカルモデル
- ローカル単体起動 → 専用推論サーバ

### 7.3 議事録モック時の軽量構成
議事録モックなど軽量デモ時には、Gemma 4 E4B を利用できるようにする。
ただし、本実装では Gemma 4 31B への置換を前提に設計する。

### 7.4 ASR方針
音声入力はGemmaのネイティブ音声ではなく、Whisper系ASRを利用する。
理由は以下の通り。

- 長尺音声に対応しやすいこと
- 議事録用途で音声入力を独立して扱えること
- モデルサイズによる精度・速度調整が可能であること

ASRはモデルサイズや実装を差し替え可能にする。

## 8. 文書取り込み方針

### 8.1 基本方針
文書は「原本の保存」と「内部表現の統一」を分けて扱う。

- 原本はそのまま保存する
- 内部では共通表現へ正規化する
- LLMは原本そのものではなく、正規化済み文書やチャンクを読む

### 8.2 原本保存
扱うファイル形式は多様であるため、原本は以下のような形式をそのまま保持する。

- PDF
- DOCX
- PPTX
- XLSX
- Markdown
- テキスト
- 画像

### 8.3 正規化
原本はサーバ側で共通内部形式へ変換する。
内部形式としては以下を保持する。

- JSON
- Markdown

JSON は構造・メタデータ・元文書との対応付け保持に用いる。Markdown は LLM 入力や可読性の高い中間表現として利用する。

### 8.4 チャンク化
正規化済み文書は、検索・比較・LLM参照のためにチャンク化する。
チャンク化では以下を考慮する。

- ページ・節・見出し構造
- 適切な長さ
- 出典追跡可能性
- 後段比較処理で使いやすいこと

---

## 9. 文書投入方式
文書投入は、利用者の運用に合わせて2系統用意する。

### 9.1 CLI投入
サーバ上の所定ディレクトリに資料を配置し、コマンドで一括変換・チャンク化・索引化を行う。

想定用途:
- 学内資料の初期一括投入
- 定期更新
- 管理者による運用

### 9.2 GUI投入
ブラウザUIから複数ファイルを一括アップロードし、そのまま変換・チャンク化・索引化まで行う。

想定用途:
- 教員による日常的な資料追加
- 論文・レポートの一時投入
- 比較対象ファイルのその場アップロード

### 9.3 内部処理の共通化
CLI投入とGUI投入は入口のみ異なり、内部では同一の処理パイプラインを利用する。

共通パイプライン:
- ingest
- normalize
- chunk
- index

## 10. チャットと文書の関係

### 10.1 永続資料参照
学内資料や校内限定資料は、永続文書基盤に登録し、チャットが検索・参照できるようにする。

### 10.2 一時文書参照
論文やレポートは、チャットセッションに紐づく一時アップロードとして扱う。

理由:
- 恒久保存が必須ではない場合がある
- 個人情報や未公開研究内容を含む場合がある
- 比較や添削など、その場限りの処理が多い

### 10.3 複数文書参照
チャットや比較ツールは、単一文書だけでなく複数文書を同時参照可能でなければならない。

対象例:
- レポートAとレポートBの類似確認
- 原稿の版差比較
- 提出物間の表現類似確認
- 学内資料と提出文書の照合

---

## 11. ツール設計
チャットから呼び出すツールは、少なくとも以下の責務で分離する。

### 11.1 文書検索ツール
- 永続資料群から該当箇所を検索する

### 11.2 文書読解ツール
- 単一文書または指定範囲を読ませる
- 要約・添削・説明を行う

### 11.3 文書比較ツール
- 2文書以上の差分や類似箇所を抽出する

### 11.4 議事録ツール
- 音声から議事録案を作る専用ワークフローを実行する

これらを曖昧に1ツールへまとめず、役割ごとに分離する。

---

## 12. 議事録機能の詳細

### 12.1 処理フロー
議事録作成は以下の手順で行う。

1. 音声録音または音声ファイルアップロード
2. ASR による raw transcript 生成
3. LLM による transcript cleanup
4. LLM による議事録案生成
5. raw / cleaned / minutes の確認

### 12.2 独立性
議事録機能は通常チャットとは別UIまたは別ワークフローで実行する。
ただし、チャットのツール選択UIから呼び出し可能にする。

### 12.3 期待する出力
- 面談名
- 実施日
- 参加者
- 主な相談内容
- 決定事項
- 今後の対応
- 注意事項
- 自由記述

---

## 13. 非機能要件

### 13.1 セキュリティ
- 校内機密データを外部クラウドへ送信しない
- ローカルまたは閉域環境で運用可能であること
- 保持データの削除が容易であること
- 将来的に認証・権限管理・監査ログを実装可能であること

### 13.2 同時利用耐性
複数教員による同時利用を想定し、以下の方針を取る。

- FastAPI などのAPI Gatewayと推論サーバを分離する
- 長い処理はジョブ化する
- チャット応答と重いバッチ処理を分ける

### 13.3 拡張性
- モデル差し替えが容易であること
- 新しいツール追加がしやすいこと
- 文書形式追加がしやすいこと
- UI を段階的に拡張できること

### 13.4 保守性
- CLI と GUI の両入口で内部パイプラインを共通化すること
- ツール責務を分離し、過剰な密結合を避けること

## 16. まとめ
本システムは、校内機密データを安全に扱うためのローカルAI支援基盤であり、中心はチャット型UIとする。
ただし、議事録のように特有のワークフローを持つ機能は専用ツールとして切り出し、チャットから呼び出せる構造にする。

文書処理については、原本保存・正規化・チャンク化・索引化を共通基盤として整備し、CLI と GUI の両方から投入可能にする。
また、学内資料の参照だけでなく、論文・レポート・提出物の読解や複数文書比較まで見据えた構造とする。

これにより、短期的には議事録モックの実現、長期的には校内ローカルAI基盤への発展を可能にする。
