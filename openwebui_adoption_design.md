# Open WebUI 採用構成案

## 1. 目的
本資料は、校内機密データ向けローカルAI支援基盤において、チャットUIとして Open WebUI を採用する場合の構成方針を整理するものである。

本構成では、Open WebUI を **前面UIおよびチャット操作基盤** として利用し、既存の要件定義・設計で整理した以下の独自機能は自前バックエンドに保持する。

- 学内資料の永続文書基盤
- 一時文書アップロード基盤
- 複数文書比較機能
- 議事録専用ツール
- 監査、保持期間、削除、権限制御などの学校向け運用制御

つまり、Open WebUI を「全部入りの業務ロジック基盤」として採用するのではなく、**ローカルチャットUI兼ツールランチャー** として採用する。

実装メモ: 現在のデモ実装では、Open WebUI を外部チャットUIとして並存させつつ、FastAPI 配信の `/chat/` に簡易チャットUIを残す。これは一時文書添付、履歴保存、ツール候補表示を自前APIで検証するための導線であり、Open WebUI 連携APIとは同じ文書基盤・比較基盤・監査ログを共有する。

---

## 2. 採用理由
Open WebUI を採用する主な理由は以下の通りである。

1. チャットUI/UXを短期間で改善できる
2. ローカル運用に適した self-hosted 前提の構成である
3. Ollama や OpenAI-compatible API に接続しやすい
4. 将来的な SGLang 等への切り替えとも整合しやすい
5. ファイル投入、Knowledge、ツール拡張などの土台が既にある

一方で、学校向けシステムとして必要な独自要件まで Open WebUI 標準に寄せると、設計の一貫性や監査性が落ちる可能性がある。
そのため、**UIは Open WebUI、業務ロジックは自前API** という分離方針を採用する。

---

## 3. 基本方針

### 3.1 Open WebUI に任せるもの
- 通常チャットUI
- 会話セッション管理
- モデル選択UI
- メッセージ表示
- Markdown表示
- ファイル添付の入口
- 専用ツールの起動導線

### 3.2 自前バックエンドに残すもの
- Docling による文書正規化
- CLI/GUI 共通の文書取り込みパイプライン
- 一時文書のセッション紐付け
- 比較・差分確認ロジック
- 議事録ワークフロー
- 削除、保持期間、監査ログ
- 学校向け権限制御

### 3.3 境界の考え方
Open WebUI は「ユーザーが使う表側」であり、
自前バックエンドは「学校要件に合わせたデータ処理と業務機能」を担う。

この分離により、将来的に Open WebUI を別UIへ置き換えても、業務ロジック資産を維持しやすくする。

---

## 4. 全体アーキテクチャ

```text
[User Browser]
   |
   v
[Open WebUI]
   | \
   |  \----> [Ollama / SGLang / OpenAI-compatible LLM Runtime]
   |
   +-------> [Local School AI Backend API]
                |
                +--> [Minutes Tool Service]
                +--> [Document Ingest Service]
                +--> [Session Document Service]
                +--> [Comparison Service]
                +--> [Search / RAG Service]
                +--> [Audit / Retention Service]
                |
                +--> [SQLite / FTS5 / Qdrant(local)]
                +--> [Original Files / Normalized Files / Chunks]
```

---

## 5. 接続方式

### 5.1 LLM 接続
Open WebUI は以下いずれかのローカルLLMランタイムに接続する。

- 開発環境: Ollama
- 本番候補: SGLang
- 将来候補: その他 OpenAI-compatible local server

このため、チャット本体で使うモデルは OpenAI-compatible API 前提で統一する。

### 5.2 自前バックエンド接続
Open WebUI から自前バックエンドへの接続は、以下のいずれかで行う。

- Action/Tool 的な専用ツール呼び出し
- 独自API を叩く軽量ブリッジ
- 議事録専用画面への導線

実装初期は、最も単純な方法として **Open WebUI から外部の自前APIへ接続する構成** を採用し、
複雑な統合は後回しにする。

---

## 6. 議事録ツールの扱い
議事録ツールは通常チャットに埋め込まない。

理由:
- 音声入力から始まる特殊フローである
- transcription -> cleanup -> minutes の段階処理が必要である
- ジョブ処理が長くなりやすい
- チャットの1発話として扱うにはUI上の整合性が悪い

### 方針
- Open WebUI から「議事録作成」ツールとして呼び出せるようにする
- 実際の処理は独立した議事録サービスで行う
- 必要なら完成した議事録だけをチャットへ戻せるようにする

つまり、**Open WebUI から呼び出せる独立専用ツール** として扱う。

---

## 7. 文書参照とアップロードの扱い

### 7.1 永続資料
対象:
- 学生生活ハンドブック
- 校則
- 学内手続き文書
- 学内限定資料

これらは自前バックエンドで管理する。
Open WebUI は検索・質問のUIとして使う。

### 7.2 一時文書
対象:
- 論文
- レポート
- 提出物
- 比較対象文書

これらも最終的な保持管理は自前バックエンドで行う。
Open WebUI は、セッションに文書を紐付ける入口として使う。

### 7.3 基本原則
- 原本はサーバ上に保存
- 正規化は自前パイプラインで行う
- LLM に直接バイナリを渡さない
- 正規化結果や chunk をツール経由で読ませる

---

## 8. 文書比較機能の位置づけ
レポートの差分確認、似ている箇所の抽出、複数文書比較は独立した比較サービスとして保持する。

Open WebUI 側では、
- 比較対象文書を指定するUI
- 比較ツール起動の導線
- 結果表示
を担う。

比較ロジック自体は自前バックエンドに置く。

---

## 9. データ保存と責務分離

### 9.1 Open WebUI 側に依存しすぎない理由
Open WebUI の内部データモデルやプラグイン仕様に強く依存すると、将来の置き換えや監査対応が難しくなる。

そのため、以下は Open WebUI に閉じ込めない。
- 文書原本
- 正規化済み文書
- chunk データ
- 学校独自メタデータ
- 監査ログ
- 削除・保持期間設定

### 9.2 自前で持つべき保存先
```text
data/
  originals/
  normalized/
  chunks/
  indexes/
  audit/
```

Open WebUI は UI 層に留め、学校向け業務データの主保存先にはしない。

---

## 10. セキュリティ方針
Open WebUI 採用時も、以下の原則を維持する。

- 機密データを外部クラウドへ送らない
- 外部SaaS依存を増やさない
- 導入する拡張は信頼できるものに限定する
- 外部の野良プラグインを安易に入れない
- 実行時に動作するコードは監査できる範囲に限定する

Open WebUI のツール拡張機構は強力だが、学校用途では **自作または十分確認済みの拡張だけを使う** 方針とする。

---

## 11. 採用時の実装優先順位

### Phase A
- Open WebUI を通常チャットUIとして導入
- Ollama 接続でチャットを成立させる
- 既存自作チャットUIは停止、退役、またはデモ・互換確認用として共存させる

### Phase B
- Open WebUI から自前バックエンドの資料参照ツールを呼ぶ
- 永続資料検索と回答生成を接続する

### Phase C
- 一時文書アップロードと読込ツールを接続する
- 複数文書比較ツールを接続する

### Phase D
- 議事録作成ツールを Open WebUI から起動可能にする
- 必要なら議事録結果をチャットへ戻す

### Phase E
- 監査ログ、削除、保持期間、権限制御を強化する

---

## 12. メリット
- チャットUI/UX を短期間で改善できる
- モデル切替をしやすい
- 将来の資料参照、文書比較、専用ツール連携の導線を確保できる
- 自前で全部作るより開発コストを抑えられる
- 自前バックエンド資産を維持できる

---

## 13. デメリットと注意点
- Open WebUI の内部仕様に寄せすぎると独立性が落ちる
- 拡張を増やしすぎると保守が難しくなる
- UI と業務ロジックの責務が混ざると設計が崩れる
- 学校用途では、導入する拡張コードの監査が必須である

---

## 14. 結論
Open WebUI は、本システムの **チャットUI兼ツールランチャー** として採用する価値が高い。

ただし、Open WebUI をシステムの中核業務ロジック基盤にはしない。
中核となるのは、あくまで自前のローカルバックエンドであり、以下を保持する。

- 学内資料基盤
- 一時文書基盤
- 比較ツール
- 議事録ツール
- 学校向け運用制御

したがって本プロジェクトでは、
**「Open WebUI を前面UIとして採用しつつ、学校固有の処理系は独立バックエンドとして維持する」**
という構成を採用する。
